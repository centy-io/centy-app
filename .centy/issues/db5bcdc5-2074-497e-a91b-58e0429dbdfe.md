# Dynamic page types driven by .centy/<type>/config.yaml

## Problem

Currently, the centy app only supports hardcoded page types (`issues`, `docs`). However, the `.centy/` directory already supports additional page types such as `personas` and `stories`, each with their own `config.yaml` describing their structure, features, statuses, and custom fields.

The app does not dynamically discover or render these additional page types, meaning users cannot navigate to, create, or manage items in any custom page type (e.g. `personas`, `stories`, or any future type).

## Goal

Make the centy app dynamically discover all page types defined under `.centy/<type>/config.yaml` and render a fully functional page for each one, driven entirely by the config.

## Proposed Solution

1. **Discovery**: At startup (and on file-system change), scan all subdirectories of `.centy/` for a `config.yaml` file. Each such directory defines a page type.

2. **Config-driven rendering**: Use the `config.yaml` to determine:
   - Page name and navigation label (`name` field)
   - Item identifier strategy (`identifier`: `uuid` or `slug`)
   - Which features to show/hide (status badge, priority, assets, orgSync, move, duplicate)
   - Available statuses and default status
   - Custom fields (name, type, required)

3. **Navigation**: Automatically add each discovered page type to the sidebar/nav.

4. **CRUD operations**: Support creating, viewing, editing, and deleting items for any discovered page type, using the config to drive the UI (e.g. render custom fields in the item form/detail view).

5. **File structure**: Items for a given type live in `.centy/<type>/<identifier>.<ext>` (matching the existing pattern).

## Example

Given `.centy/personas/config.yaml`:

```yaml
name: persona
identifier: slug
features:
  displayNumber: false
  status: false
  priority: false
  assets: false
  orgSync: false
  move: false
  duplicate: false
statuses:
  - active
  - archived
defaultStatus: active
customFields:
  - name: role
    type: string
    required: false
  - name: goals
    type: string
    required: false
  - name: pain-points
    type: string
    required: false
```

The app should render a "Personas" page in the nav, list all files from `.centy/personas/`, and provide a creation form with `role`, `goals`, and `pain-points` fields.

## Acceptance Criteria

- [ ] App discovers all `.centy/<type>/config.yaml` files on startup
- [ ] Each discovered type appears as a page in navigation
- [ ] Items in each type directory are listed and viewable
- [ ] Creating a new item uses config-driven form (custom fields, statuses)
- [ ] Features (status badge, priority, assets, etc.) are shown/hidden per config
- [ ] Existing `issues` and `docs` pages continue to work as before (no regression)
- [ ] Works for `personas` and `stories` as concrete examples
