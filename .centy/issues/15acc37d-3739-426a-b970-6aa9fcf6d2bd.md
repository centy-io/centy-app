# Consolidate font colors to use design system tokens (single source of truth)

## Problem

Many component CSS files use hardcoded hex color values instead of the design system tokens defined in `styles/globals.css`. This means font colors don’t properly adapt between light and dark modes.

For example, `.project-name` in `styles/globals.css` uses `color: #fff` which is hardcoded for dark mode and breaks in light mode.

### Files with hardcoded font colors (bypassing design tokens):

- `styles/globals.css` (components section like `.project-selector-*`, `.daemon-*`)
- `styles/components/ProjectSelector.css` — ~35 hardcoded colors
- `styles/components/OrgSwitcher.css` — ~10 hardcoded colors
- `styles/components/DaemonDisconnectedOverlay.css` — ~12 hardcoded colors
- `styles/components/DaemonStatusIndicator.css` — ~6 hardcoded colors
- `styles/components/MobileNotSupportedOverlay.css` — ~8 hardcoded colors
- `styles/components/MarkdownEditor.css` — ~8 hardcoded colors
- `styles/components/TextEditor/TextEditor.css` — ~8 hardcoded colors
- `styles/components/DemoModeIndicator.css` — ~2 hardcoded colors

## Expected behavior

All font colors should use the semantic tokens defined in the design system:

- `var(--color-text-primary)` for main text
- `var(--color-text-secondary)` for supporting text
- `var(--color-text-muted)` for subtle/disabled text
- `var(--color-gray-*)` scale for other shades

These tokens already handle dark/light mode switching automatically via the `[data-theme='dark']` override in `globals.css`.

## Steps

1. Audit all CSS files for hardcoded `color: #...` values
1. Map each hardcoded value to the closest semantic token
1. Replace hardcoded values with `var(--color-*)` references
1. Verify both light and dark modes render correctly after changes
