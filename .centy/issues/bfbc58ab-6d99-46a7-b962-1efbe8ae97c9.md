# Two-column detail layout: main content left, metadata and links sidebar right

## Problem

All three detail views (`GenericItemDetail`, `IssueDetail`, `OrgIssueDetail`) use a single-column layout capped at `800px` centered on the page. Metadata (status, priority, assignees, dates, custom fields) and links are stacked inline below the title and above or after the description, forcing the user to scroll past relational context to read content and vice versa.

Concretely:

- On wide screens the 800px column wastes ~40–50% of the horizontal viewport
- While reading a long description or comment thread, metadata scrolls out of view — there is no persistent summary of the item's state
- Links (the most relational piece of context) sit at the very bottom, below assets and before comments — far from where a reader's attention lands first

## Goal

Replace the single-column layout with a two-column layout across all detail views:

- **Left (main) column** — title, description/body, assets, comments (~65–70% of width)
- **Right (sidebar) column** — metadata block + links block (~30–35% of width, sticky)

The sidebar makes state and relationships permanently visible without requiring a scroll, matching the established UX pattern of tools like Linear, GitHub Issues, and Notion.

## Layout Spec

**Container width:** increase max-width from `800px` to `~1100px` to give both columns breathing room.

**Column grid:** `grid-template-columns: 1fr 280px` (or `minmax(0, 1fr) 280px`) with a `~28px` gap.

**Sidebar stickiness:** `position: sticky; top: 24px; max-height: calc(100vh - 48px); overflow-y: auto;` so it stays visible when the main column is taller than the viewport.

**Responsive breakpoint:** at `≤ 900px`, collapse to single column — sidebar flows below main content (not above, to preserve the reading order: title → body → context).

**Edit mode:** the two-column structure applies in both view and edit modes. The sidebar metadata fields become their interactive controls (status dropdown, priority selector, assignee picker) — they were already rendered at the top of the content flow, they simply move right.

## Sidebar Content (in order)

### Properties section

- Status (dropdown when editable, badge when read-only)
- Priority (badge)
- Assignees (issue views only)
- Created at / Updated at (compact, de-emphasized)
- Custom fields (key: value rows)
- Tags (if present)

### Relations section

- `LinkSection` / `LinkGroupList` (grouped by link type, with add-link affordance)

Assets stay in the main column — they are content artifacts, not metadata.

## Approach

1. **Introduce `DetailLayout` shared component** (`components/shared/DetailLayout/DetailLayout.tsx`) — a thin layout wrapper that accepts `main` and `sidebar` slots and owns the CSS grid. Keeps the structural change DRY across all three views. CSS lives in `styles/components/DetailLayout.css`.

2. **Update `GenericItemDetail`** — wrap `DetailBody`'s contents in `DetailLayout`. Extract `ItemMetadata` and `LinkSection` from `GenericItemView` into the `sidebar` slot. Remove the old per-view `max-width` / `margin: 0 auto` from `GenericItemDetail.css`.

3. **Update `IssueDetail`** — wrap `IssueDetailBody` in `DetailLayout`. Move `Metadata` (status, priority, assignees) and `ViewContent`'s `LinkSection` into the `sidebar` slot. The main slot keeps title, description (`TextEditor`), assets, and `CommentThread`.

4. **Update `OrgIssueDetail`** — wrap `OrgIssueReadView` in `DetailLayout`. Move `ItemMetadata` into the `sidebar` slot.

5. **Section headers in sidebar** — add lightweight section labels ("Properties", "Relations") above each block to give the sidebar visual structure.

6. **Responsive CSS** — in `DetailLayout.css`, at `≤ 900px` reset grid to single column and remove sticky positioning.

7. **Remove now-redundant inline metadata layout** — the old inline horizontal badge row for metadata no longer needs to be in the content flow; adjust or remove its containing wrapper.
