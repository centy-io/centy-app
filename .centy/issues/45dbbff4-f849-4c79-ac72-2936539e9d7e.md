# Cache daemon update check in localStorage to avoid GitHub API rate limiting

## Problem

The `DaemonUpdateBadge` calls `fetchLatestDaemonVersion()` on every daemon status check interval, hitting `api.github.com/repos/centy-io/centy-daemon/releases/latest` directly from the browser. GitHub's unauthenticated API is rate-limited to 60 requests/hour per IP, so the call quickly returns 403, `latestDaemonVersion` stays `null`, and the badge never appears — even when an update is genuinely available (confirmed: daemon is on `0.9.3`, latest is `0.10.0`).

## Goal

Cache the result of the GitHub version check in `localStorage`, keyed by the running daemon version. Only re-fetch from GitHub when the stored daemon version differs from the current one (i.e. after the user has updated the daemon). This way the API is called once per daemon version, not on every polling interval.

## Approach

1. Add `lib/daemonUpdateCache.ts` — read/write helpers for `localStorage` with shape `{ daemonVersion: string, hasUpdate: boolean }`
2. In `useCheckDaemonStatus` (`DaemonStatusProvider.hooks.helpers.tsx`): after getting `daemonInfo.version`, read the cache. If the stored version matches → use cached `hasUpdate`. Otherwise → fetch GitHub, compute `hasUpdate = isNewerVersion(current, latest)`, write cache.
3. Replace `latestDaemonVersion: string | null` in the context with `daemonUpdateAvailable: boolean` — no version string needs to be surfaced to the UI.
4. Update `DaemonUpdateBadge` to use `daemonUpdateAvailable` directly (remove the `isNewerVersion` call from the component).
5. Update `DaemonUpdateDialog` to remove version display (just show the install command).
6. Update all affected specs.
