# Support opening workspace from item using slug ID

## Problem

Currently, the `OpenInTempWorkspaceRequest.issue_id` field accepts a UUID or display number to identify the item when opening a workspace. There is no support for using the item’s slug ID.

## Goal

Allow opening a workspace by passing the item’s slug ID (e.g., `my-feature-request`) in addition to UUID and display number. This gives users a more human-readable way to reference items when opening workspaces.

## Implementation

### Daemon (proto + backend)

- Update `OpenInTempWorkspaceRequest.issue_id` field comment to document that slug ID is accepted
- Update the daemon’s item resolution logic to try matching by slug ID when the provided value is not a UUID and not a pure number
- Same change for `OpenAgentInTerminalRequest.issue_id` and any other RPCs that resolve items by ID

### Frontend

- No frontend changes expected unless we add a UI surface that passes slug IDs (the backend change is transparent)

## Acceptance Criteria

- [ ] Passing a slug ID to `OpenInTempWorkspaceRequest.issue_id` resolves the correct item and opens the workspace
- [ ] UUID and display number continue to work as before
- [ ] `OpenAgentInTerminalRequest` also supports slug ID resolution
- [ ] Proto field comments are updated to reflect the three accepted formats
