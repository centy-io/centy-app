# Migrate IssuesList to GenericItemsList pattern

# Migrate IssuesList to GenericItemsList Pattern

## Goal

Replace the bespoke `IssuesList` component with the `GenericItemsList` framework so issues are rendered through the same pipeline as all other item types. This reduces duplication, makes issues benefit from future generic improvements, and removes a parallel code path that must be maintained separately.

---

## Current State

| Aspect        | IssuesList                                   | GenericItemsList                            |
| ------------- | -------------------------------------------- | ------------------------------------------- |
| Location      | `components/issues/IssuesList/`              | `components/generic/GenericItemsList.tsx`   |
| Display       | Tanstack Table (sortable/filterable columns) | Card grid                                   |
| Item type     | Hardcoded `"issues"`                         | Dynamic `itemType` prop                     |
| Delete        | None in list view                            | Soft delete + hard delete with confirmation |
| Filtering     | Per-column filters (text + multi-select)     | None                                        |
| Context menu  | Pin / View / Move / Duplicate                | None                                        |
| Priority      | Dedicated column with custom sort            | Not supported                               |
| Workspace     | New Workspace button + modal                 | Not supported                               |
| Last seen     | Dedicated column                             | Not supported                               |
| Config-driven | No                                           | Yes (ItemTypeConfigProto)                   |

## Migration Phases

### Phase 1 — Add table view to GenericItemsList

The card grid is the only display format in GenericItemsList. Issues need a table layout. Add an optional table mode driven by item type config.

- Add a `view: "table" | "grid"` field to `ItemTypeConfigProto` (or derive from features flags)
- Create `GenericItemsTable.tsx` — a Tanstack Table wrapper similar to `IssuesTable.tsx` but column-agnostic
- Create `useGenericItemsTable.ts` hook that builds columns from item type config (standard fields: #, title, status; optional: priority, dates)
- Column definitions driven by `config.features.*` flags rather than hardcoded
- Plug into `GenericListContent.tsx` as an alternative to `ItemsGrid`

### Phase 2 — Generalize column filtering and sorting

`IssuesList` has per-column filter UI (text inputs for #/title, multi-select dropdowns for status/priority). This needs to become a generic capability.

- Create `GenericColumnFilter.tsx` — renders the appropriate filter control based on column type
- Support filter types: `text`, `select` (for status/priority), `date-range`
- Add `useGenericTableSettings.ts` hook for persisting sort/filter state (mirrors `useIssueTableSettings`)
- Wire into `GenericItemsTable`

### Phase 3 — Generalize context menu (Pin / Move / Duplicate)

`IssuesList` has a right-click context menu with pin, view, move, and duplicate actions. These are not issues-specific and should be available to any item type.

- Create `useGenericContextMenu.ts` — generalizes `useIssueContextMenu` with item type awareness
- Create `GenericContextMenu.tsx` — renders the context menu UI
- Move/Duplicate modals already exist (`MoveModal`, `DuplicateModal`); make them accept generic item type
- Add `useGenericMoveActions.ts` — generalizes `useIssueMoveActions`
- Wire into `GenericItemsList`

### Phase 4 — Priority as a generic concept

`IssuesList` has a dedicated Priority column with P1/P2/P3 normalization, custom sort, and filter options. Priority should become a first-class generic feature.

- Add `priority` to `ItemTypeConfigProto.features` (already may exist as a flag — verify)
- Create `genericPriorityColumn.tsx` — reusable priority column definition
- Move `getIssuePriorityClass` → `getPriorityClass` utility (generic)
- Move `IssuesPriorityOptions` → `PriorityOptions` (generic)

### Phase 5 — Standalone workspace support

The `IssuesList` header has a "New Workspace" button that opens `StandaloneWorkspaceModal`. Evaluate whether this is issues-only or should be generic.

- If issues-only: keep `StandaloneWorkspaceModal` as an issues-specific overlay injected into `GenericListHeader` via a prop slot
- If generic: add workspace feature flag to config and wire into `GenericListHeader`

### Phase 6 — Last seen tracking

`IssuesList` tracks and displays when each issue was last viewed. This is currently issues-specific.

- Evaluate if `useLastSeenIssues` should become `useLastSeenItems` (keyed by item type)
- Add `lastSeen` as an optional column in `GenericItemsTable` driven by a config feature flag

### Phase 7 — Migrate issues page to GenericItemsList

Once all capabilities are in place:

- Remove `IssuesList` component and all sub-components (`IssuesHeader`, `IssuesContent`, `IssuesTable`, `IssuesModals`, `IssuesColumnFilter`, `IssuesTableHeader`)
- Remove all issues-specific hooks (`useIssuesData`, `useIssuesTable`, `useIssueContextMenu`, `useIssueMoveActions`)
- Update `/app/[organization]/[project]/issues/page.tsx` to render `<GenericItemsList itemType="issues" />`
- Migrate `styles/pages/IssuesList.css` into `styles/pages/GenericItems.css`
- Update all tests to use the generic spec utilities

## Key Design Decisions

1. **Table vs grid:** Issues stay as a table. Config flag (`view: "table"`) selects table mode in `GenericItemsList`.
2. **Priority:** Promote to generic feature flag; column renders when `config.features.priority === true`.
3. **Context menu:** Generalize fully — any item type with move/duplicate enabled in config gets the menu.
4. **Delete pattern:** Adopt GenericItemsList soft/hard delete in the migrated issues list.
5. **Last seen:** Generalize as `useLastSeenItems(itemType)` keyed by type.
6. **Workspace:** Keep as issues-specific injected slot initially; revisit when workspace feature is broader.
